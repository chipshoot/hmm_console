⏺ Deploying Flutter App to Personal iPhone (Production)

  Prerequisites

  1. Apple ID enrolled in the Apple Developer Program (or free account for 7-day
   provisioning)
  2. Xcode installed and updated
  3. iPhone connected via USB (or configured for wireless debugging)

  Steps

  1. Configure signing in Xcode

  open ios/Runner.xcworkspace

  In Xcode:
  - Select Runner project → Signing & Capabilities
  - Set Team to your Apple ID / Developer account
  - Change Bundle Identifier to something unique (e.g., com.yourname.hmmconsole)
  - Ensure "Automatically manage signing" is checked

  2. Trust your device

  On your iPhone: Settings → General → VPN & Device Management → trust your
  developer certificate (appears after first install attempt).

  3. Verify production API config

  Let me check your current API configuration to make sure it points to
  production.

  Read 3 files (ctrl+o to expand)

⏺ Your app uses the API_ENV compile-time flag to switch environments. Both
  ApiClient and IdpTokenService default to development. Here's how to deploy to
  your iPhone targeting production:

  4. Build & run on your iPhone in production mode

  Connect your iPhone via USB, then run:

  flutter run --release --dart-define=API_ENV=production

  This will:
  - Set API_ENV=production, pointing to https://api.homemademessage.com and
  https://idp.homemademessage.com
  - Build in release mode (optimized, no debug banner)
  - Deploy to your connected iPhone

  If you just want to debug against production:

  flutter run --dart-define=API_ENV=production

  This runs in debug mode (hot reload, DevTools) but talks to the production
  APIs.

❯ can I use personal iphone as simulator device without pay apply for developer 
 account?                                                                       

⏺ Yes, you can deploy to a personal iPhone for free without a paid Apple
  Developer account ($99/year). Here's what you need:

  Free Apple ID Signing

  1. Sign in to Xcode with your regular Apple ID (Settings → Accounts → add your
   Apple ID)
  2. Open ios/Runner.xcworkspace in Xcode
  3. Under Runner → Signing & Capabilities, select your Personal Team and set a
  unique Bundle ID
  4. Xcode will create a free provisioning profile automatically

  Limitations of Free Signing

  - App expires after 7 days — you must re-deploy from Xcode/Flutter after that
  - Max 3 apps on your device at a time (per Apple ID)
  - No push notifications, App Groups, or some entitlements
  - No TestFlight or App Store distribution
  - Must trust the certificate on your iPhone each time: Settings → General →
  VPN & Device Management

  Deploy Steps

  # Open Xcode to set up signing first
  open ios/Runner.xcworkspace

  # Then run to your iPhone
  flutter run --release --dart-define=API_ENV=production

  If you hit a "device not registered" or provisioning error, just opening the
  project in Xcode and hitting the Play button once will trigger the automatic
  profile creation. After that, flutter run will work from the command line.

  Bottom line: A paid developer account is only needed for App Store /
  TestFlight distribution and advanced capabilities. For personal testing on
  your own device, the free Apple ID is sufficient.

  Alternatives to Avoid Apple's $99/year

  Option 1: Web App (PWA) — Cheapest

  - Wrap your Flutter app as a Progressive Web App
  - Flutter supports flutter build web out of the box
  - Host on Firebase Hosting (free tier), Vercel, or Cloudflare Pages
  - Users "install" it from the browser (Add to Home Screen)
  - Cost: $25 (Google Play only) + free hosting
  - Downside: No App Store presence for iOS, limited native APIs (no push on iOS
   Safari until recently, no background tasks)

  Option 2: Android on Play Store + iOS via Web

  - Publish Android app to Google Play ($25 one-time)
  - Serve iOS users a PWA version
  - Best budget compromise if most users are on Android

  Option 3: Sideloading (Not Recommended for Real Users)

  - Android: Distribute APK directly (no store needed, free)
  - iOS: Not practical — free certs expire in 7 days, max 3 devices

  1. User trust — people expect apps from official stores
  2. Auto-updates — critical for bug fixes
  3. TestFlight — free beta testing for up to 10,000 users
  4. Push notifications — requires Apple Developer account on iOS
  5. $99/year is less than $9/month — cheaper than most hosting services

  Backend Costs (Your Existing Setup)

❯ is firebase retiring in next year?                                            

⏺ Web Search("Firebase retiring shutdown 2026 2027")
  ⎿  Did 1 search in 7s

⏺ No — Firebase itself is NOT retiring. Only Firebase Studio (a cloud-based
  IDE/dev environment) is shutting down.

  What's Shutting Down

  Firebase Studio (the code editor, not the backend services):
  - June 2026: No new workspaces can be created
  - March 2027: Fully shut down

  Google is shifting that IDE functionality to Google AI Studio and Antigravity.

  What's NOT Affected

  All the core Firebase services your app uses are continuing normally:
  - Firebase Authentication
  - Cloud Firestore
  - Cloud Functions
  - Firebase Hosting
  - Crashlytics
  - Analytics
  - Cloud Messaging (push notifications)

  These are core Google Cloud infrastructure — billions of apps depend on them.
  There's no indication they're going anywhere.

  Bottom Line

  Your current setup using Firebase Auth is safe. No migration needed. The
  "Firebase shutdown" headlines are about a dev tool, not the platform itself.

  ├───────────────┼──────────────────────┼─────────────────────────────────┤
  │ Complexity    │ Two auth systems to  │ One unified system              │
  │               │ maintain             │                                 │
  ├───────────────┼──────────────────────┼─────────────────────────────────┤
  │ Dependency    │ Tied to Google       │ Self-hosted                     │
  └───────────────┴──────────────────────┴─────────────────────────────────┘

  Why NOT Firebase Auth for Your Case

  1. You already have a working IDP — adding Firebase Auth means maintaining two
   auth systems
  2. Your API expects your IDP's tokens — Firebase tokens would need a separate
  exchange flow or your API would need to accept both
  3. Double dependency — if Firebase has an outage, your users can't log in even
   if your servers are fine
  4. No real benefit — Firebase Auth's value is saving you from building auth.
  You've already built it.

  What to Do

  Remove Firebase Auth dependency and keep Firebase only for services you
  actually need (Crashlytics, Analytics — these don't require Firebase Auth).

  In main.dart, you can remove Firebase.initializeApp() entirely if you're not
  using any Firebase services, or keep it only for analytics/crashlytics.

  Your current IdpConfig already has clean production/development separation:
  - Production: https://idp.homemademessage.com
  - Development: localhost:5001

  One Thing to Add

  If you want social login (Google Sign-In, Apple Sign-In), implement it on your
   IDP server side rather than through Firebase. Most IDP frameworks
  (IdentityServer, Keycloak, etc.) support this natively.

  Bottom line: You've already done the hard work. Using Firebase Auth on top
  would just add complexity with no real gain for a free app with your own
  backend.
